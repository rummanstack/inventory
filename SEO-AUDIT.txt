2. PRERENDERING  --  GOOD, WITH ONE REAL DEPLOYMENT RISK
================================================================================

frontend/scripts/prerender-public-pages.mjs is a genuine full-HTML
prerenderer, not a stub:

- Starts a local static file server over the built dist/
  (createStaticServer, lines 56-88).
- Launches headless Chrome/Edge via puppeteer-core (lines 110-114).
- For every route in PUBLIC_ROUTES (from public-routes.mjs), navigates with
  waitUntil: 'networkidle0', waits for <main> to render (line 122), then
  captures document.documentElement.outerHTML AFTER React has hydrated and
  SeoManager has run (line 123) -- so the emitted HTML already contains the
  per-route title, meta tags, canonical link, and JSON-LD.
- Writes each route to dist/<route>/index.html (writeRouteHtml, lines 90-99),
  e.g. dist/features/retail-pos/index.html.

Coverage: frontend/scripts/public-routes.mjs (lines 1-41) lists 37 routes --
/, /landing, all 10 /features/*, all 6 /solutions/*, all 12 /software/*, plus
/pricing, /contact, /get-started, /founder, /privacy-policy, /terms.
Cross-checked against:
  - frontend/src/features/landing/data/seoPages.js  (10 featurePages, 6 solutionPages)
  - frontend/src/features/landing/data/intentPages.js (12 intentPages)
Counts match exactly -- every real content page is covered. Also cross-checked
against frontend/src/app/App.jsx:144-159 (the actual public <Route> entries)
-- no orphaned public route exists outside this list. Private app routes
(frontend/src/app/routes.js, APP_ROUTES) are correctly excluded.

Wiring:
  - frontend/package.json:8
    "build": "vite build && node scripts/prerender-public-pages.mjs"
    Prerendering runs on every production build.
  - root scripts/render-build.mjs:42 runs `npm run build` inside frontend/.
  - vercel.json: "buildCommand": "npm run render-build"
  - frontend/render.yaml: buildCommand: npm run render-build
  - backend/app.js:38-44 serves express.static(staticRoot) BEFORE the SPA
    fallback catch-all, so a request to e.g. /features/retail-pos resolves
    to the prerendered index.html in that directory, not the generic SPA
    shell -- prerendering is actually effective for crawlers.

RISK: findBrowserExecutable() (prerender-public-pages.mjs:24-35) only checks
PUPPETEER_EXECUTABLE_PATH and four hardcoded WINDOWS Chrome/Edge paths
(lines 16-22). Render and Vercel build on Linux containers. The repo has no
Dockerfile/Aptfile/nixpacks config that installs Chromium, and puppeteer-core
(a devDependency, frontend/package.json:23) does not bundle a browser.
Unless PUPPETEER_EXECUTABLE_PATH is set to a real Linux Chromium binary in
the Render/Vercel dashboard env vars (not visible in the repo -- unverifiable
from here), this build step should throw "No Chrome or Edge executable was
found" and either fail the whole deploy, or (if failures are silently
tolerated somewhere) skip prerendering, leaving crawlers with only the
generic SPA shell in index.html.
ACTION: verify directly against the live deployment's build logs / env vars.

9. audit-public-navigation.mjs  --  PARTIAL (useful, but narrow and not in CI)
================================================================================

frontend/scripts/audit-public-navigation.mjs (run via `npm run
audit:navigation`, frontend/package.json:14) is a post-build internal-linking
checker, not a general SEO checker:

- Reads the PRERENDERED HTML for every route in INDEXABLE_PUBLIC_ROUTES
  (lines 57-58) and extracts href="..." links pointing to other known public
  routes (collectLinks, lines 40-52).
- Computes:
    - orphan routes (zero incoming internal links, line 70)
    - "weak" routes (fewer than 2 incoming links, line 71)
    - detail pages (/features/:slug, /solutions/:slug) missing a link back
      to their parent hub (lines 72-81)
- Writes a plain-text report to frontend/public-navigation-audit.txt
  (lines 11, 108-109) -- it does NOT fail the build or assert anything; it
  is purely diagnostic output for a human to read.

What it does NOT check: title/description uniqueness, canonical correctness,
structured data, heading hierarchy, or image alt text -- it only validates
internal link topology among already-prerendered pages.

It is also not wired into package.json's build script (only `vite build &&
prerender-public-pages.mjs` runs automatically, per package.json:8) --
audit:navigation (package.json:14) is a separate, manually-invoked script.
Nothing currently enforces "every route is linked from somewhere" as part of
CI/build; it must be run and read manually.

Verdict: a solid answer to "does internal linking make the programmatic
pages reachable," but it is not a general SEO auditor and isn't automated.

ADDENDUM (fixed same day, after the audit above)
================================================================================

Three fixes were implemented following this audit:

1. Swapped puppeteer-core -> puppeteer (frontend/package.json) and removed
   the hardcoded Windows-only Chrome/Edge executable paths in
   prerender-public-pages.mjs. Full puppeteer bundles/downloads its own
   Chromium per-platform at npm install time, so this now works on
   Render/Vercel's Linux build containers without any dashboard env var.

2. Added frontend/scripts/generate-sitemap.mjs, generating sitemap.xml from
   public-routes.mjs (single source of truth) instead of hand-maintaining it.
   Wired into frontend/package.json's build script to run before `vite build`.

3. Wired audit-public-navigation.mjs into the build script to run
   automatically after prerendering (it only writes a diagnostic report,
   never fails the build, so this is zero-risk).

A FOURTH, more serious bug was discovered while verifying these fixes
locally: root scripts/render-build.mjs (the actual script Render/Vercel run
via `npm run render-build`) was deleting the fresh build output in
backend/public/dist and replacing it with a copy from frontend/dist -- a
directory Vite no longer writes to (vite.config.js's build.outDir points
directly at ../backend/public/dist). frontend/dist is gitignored and would
not exist on a clean CI checkout, so this copy step likely either crashed
the deploy or silently served stale content. Fixed by having render-build.mjs
mirror the real build output (backend/public/dist) into the root /public
folder Vercel needs, instead of copying from the wrong/stale source.

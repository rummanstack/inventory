INVENTORY APPLICATION PERFORMANCE IMPROVEMENT PLAN
==================================================

Recommended implementation order
--------------------------------

1. Optimize images

The production build contains many PNG files between approximately 1 MB and
2.6 MB. Convert large images to WebP or AVIF, generate responsive image sizes,
and lazy-load images below the fold. This will likely provide the largest
improvement for public and landing pages.


2. Lazy-load translations

Both translation files are currently imported during application startup:

- English: approximately 254 KB
- Bangla: approximately 471 KB

Load only the selected language using dynamic imports. This should reduce the
current main JavaScript bundle, which is approximately 682 KB before gzip.


3. Introduce TanStack Query gradually

Use TanStack Query for API/server data to:

- Cache API responses.
- Deduplicate identical requests.
- Configure suitable stale times.
- Invalidate related queries after successful mutations.
- Disable refetching on browser-window focus.
- Standardize loading and error states.

Keep React Context for authentication, tenant information, permissions, theme,
language, notifications, and confirmation dialogs. Remove the custom shared-
data invalidation mechanism after the relevant API data has been migrated.


4. Split the global React context

frontend/src/app/useInventoryApp.jsx currently provides one large context
value. Updating products can therefore rerender components that only require
authentication, theme, or permissions.

Split it into:

- Authentication and tenant context
- UI settings context
- Confirmation and notification context
- Server data managed by TanStack Query


5. Profile and optimize expensive backend endpoints

The finance dashboard in backend/services/financeDashboardService.js executes
many aggregate queries for a single request. Consolidate related totals into
fewer SQL queries or introduce a short-lived dashboard cache after measuring
the endpoint.

The installment customer statement in
backend/services/installmentPlanService.js has an N+1 query pattern: it loads
the schedule separately for every plan. Replace this with one query that loads
schedules for all required plan IDs.


6. Optimize PostgreSQL based on measurements

Run EXPLAIN (ANALYZE, BUFFERS) on slow queries before creating new indexes.
Inspect sequential scans, repeated loops, expensive sorts, disk reads, and
incorrect row estimates.

Likely opportunities include:

- Composite indexes beginning with tenant_id.
- Indexes matching common date filters and sort order.
- pg_trgm indexes for contains searches using ILIKE and %term%.
- Keyset pagination instead of deep OFFSET pagination.
- Composite (tenant_id, created_at DESC) indexes for large logs.


7. Improve static asset delivery

When serving production assets:

- Enable Brotli or gzip at the CDN or reverse proxy.
- Cache hashed /assets/* files for one year with immutable caching.
- Keep index.html uncached or give it a short cache lifetime.
- Serve large public images through a CDN when practical.


8. Add performance monitoring

Measure performance before and after every optimization. Track:

- Frontend Core Web Vitals
- API p50, p95, and p99 response times
- Slow PostgreSQL queries
- Database connection-pool wait time
- Error rate and request volume
- JavaScript bundle and route-chunk sizes


Recommended first implementation
--------------------------------

Start with these three changes because they offer strong benefits with
relatively low architectural risk:

1. Optimize and resize large images.
2. Dynamically load only the selected translation bundle.
3. Replace the installment customer-statement N+1 queries with one batch query.

Refactor Roadmap

Phase 1
Stabilize structure before adding more features.

1. Standardize feature folders.
Every feature should follow one pattern:
- pages/
- components/
- hooks/ or viewmodels/
- services/
- utils/
- constants/

2. Kill duplicate content patterns.
Use one translation path only:
- all UI text through i18n
- no separate copy systems
- no hardcoded landing text unless truly static and non-translated

3. Define ownership rules.
Decide clearly:
- UI components render
- hooks/viewmodels prepare data
- services talk to APIs
- domain helpers calculate business rules

4. Clean naming.
Unify inconsistent names like:
- retailCustomer vs customer
- financeAccounts vs accounts
- quickSale vs sales
This matters a lot as the app grows.

Phase 2
Move business logic away from UI.

5. Extract domain logic.
Create dedicated logic layers for:
- inventory movement
- due calculation
- settlement calculation
- supplier balance logic
- finance account movement
- accounting report preparation

6. Audit component files.
Any component doing more than rendering and light interaction should be simplified.
If a component contains:
- transformation logic
- multi-step calculations
- permission branching
- reporting math
move that out.

7. Centralize shared calculations.
Expand shared utilities into domain-focused helpers so the same formula is not reimplemented in multiple places.

Phase 3
Reduce future breakage.

8. Split locale files.
Break en.js and bn.js into modules like:
- landing
- auth
- dashboard
- inventory
- sales
- finance
- reports
- settings

9. Add import discipline.
Prevent random cross-feature imports where possible.
A feature should not depend directly on internals of another feature unless intentional.

10. Build shared UI boundaries.
Make it clear what belongs in:
- global shared UI
- feature-shared UI
- page-only UI

Phase 4
Prepare for accounting-grade complexity.

11. Separate operational data from accounting data concepts.
Even if they are connected, the code should distinguish:
- business event
- financial effect
- report output

12. Introduce domain models mentally, even if lightweight.
Examples:
- sale
- purchase
- stock movement
- due ledger entry
- settlement
- expense entry
- finance transfer

13. Strengthen permission architecture.
Permissions should be centralized and traceable, not scattered in UI conditions.

Phase 5
Quality and maintainability.

14. Add focused tests around core business logic.
Start with:
- due calculation
- stock updates
- settlement math
- profit calculation
- finance balances

15. Add architecture notes.
Short docs are enough:
- folder conventions
- naming conventions
- where translations go
- where business logic goes
- how new features should be added

16. Review big pages one by one.
Pages with the highest business risk should be cleaned first:
- sales
- purchase receive
- due collection
- finance dashboard
- ledger / P&L / balance sheet
- evening settlement

Priority Order
1. Translation/content consistency
2. Feature folder standardization
3. Business logic extraction from UI
4. Locale file splitting
5. Naming cleanup
6. Permission centralization
7. Tests for financial and stock logic

What not to do
- Do not try a huge rewrite
- Do not refactor everything before shipping
- Do not mix visual redesign and domain refactor in the same pass

Best strategy
Refactor while touching active areas:
- if you edit a feature, bring that feature up to the new structure
- do not randomly churn untouched areas
